## 2026-06-13 - [Fix Silent Swallowing of Exceptions During Resource Disposal]
**Vulnerability:** Structured concurrency `DisposeResources` ignored/swallowed all exceptions thrown by resource `Dispose()` or `DisposeAsync()` calls instead of aggregating them.
**Learning:** Silently failing to clean up resources (which could result in connection leaks, unreleased locks, file descriptor leaks) poses a Denial of Service or security impact depending on what resource failed to clean up.
**Prevention:** Always accumulate disposal exceptions during teardown and throw an `AggregateException` to ensure developers are aware of cleanup failures and can handle them appropriately.

## 2026-06-14 - [FALSE POSITIVE] Linked CancellationTokenSource in StructuredRaceJob is NOT a leak — do not re-raise
**Status:** Reviewed and rejected. Do NOT open PRs claiming this is a vulnerability. Originally proposed in PR #124 ("🛡️ Sentinel: [HIGH] Fix CancellationTokenSource memory leak in StructuredRaceJob") and rejected as a false positive after review.
**Claim:** `StructuredRaceJob` creates `innerCancellationTokenSource = CancellationTokenSource.CreateLinkedTokenSource(...)` and never disposes it. The theory was that the parent `Context.CancellationTokenSource` keeps the linked source's callback registered, so over many evaluations the registrations accumulate unboundedly, causing a memory-exhaustion Denial of Service.
**Why this is not a vulnerability:** The parent is NOT long-lived. `Context.CancellationTokenSource` is shared by all overlapping/nested evaluations and refcounted by evaluation depth: `Context.EnterEvaluationScope` creates it only when none exists (the depth-0 entry) and `Context.ExitEvaluationScope` nulls it only when the depth returns to 0 — i.e., when the last evaluation in the tree exits. Its lifetime is therefore bounded by one root evaluation tree, not the process. While a parent source is alive, each `ConcurrentRace` in that tree creates a linked source from it; once the tree unwinds to depth 0 the parent field is nulled, and at that point the parent and every linked source created during the tree reference only each other — the owning `StructuredRaceJob` instances and their tasks are already garbage. That is an unreachable object island, and the .NET tracing GC reclaims islands regardless of internal cross-references, so they are all collected together. The number of live linked sources is bounded by the size of a single evaluation tree and released when it completes; there is no unbounded growth across evaluations. The classic linked-CTS registration leak requires a parent token that lives for the process/request lifetime, which never happens here.
**Guidance for future scans:** Do not flag `CreateLinkedTokenSource` usages (or "undisposed `CancellationTokenSource`") whose parent is the shared, refcounted `Context.CancellationTokenSource`. Its lifetime is bounded by a single root evaluation tree (the source is created at evaluation depth 0 and nulled when depth returns to 0), and the linked sources created during that tree form a GC-collectable island once it unwinds. Passing such a source to `IStructuredResourceGroup.AddResource(...)` for deterministic disposal is acceptable as minor hygiene, but it is NOT a HIGH-severity issue and must not be reported as a memory leak or DoS.

## 2024-05-18 - [Fix TOCTOU resource leak in StructuredResourceGroup]
**Vulnerability:** `StructuredResourceGroup.AddResource` allowed adding resources even after `DisposeResources` had gathered the items to dispose. The concurrent task running the structured job body could add resources during teardown, which would not be cleaned up, leading to a Time-of-Check to Time-of-Use (TOCTOU) Denial of Service (DoS) vulnerability.
**Learning:** The cleanup logic and additions are not strictly ordered. Adding items after or during teardown to the teardown group silently leaks unmanaged resources. Unchecked nulls might also be added.
**Prevention:** Add a `isDisposed` flag modified under lock to prevent adding resources after the teardown process begins, throwing `ObjectDisposedException` and ensuring null inputs throw `ArgumentNullException`.
