## 2026-06-14 - [FALSE POSITIVE] Linked CancellationTokenSource in StructuredRaceJob is NOT a leak — do not re-raise
**Status:** Reviewed and rejected. Do NOT open PRs claiming this is a vulnerability. First proposed in PR #124 ("🛡️ Sentinel: [HIGH] Fix CancellationTokenSource memory leak in StructuredRaceJob") and closed after review.
**Claim:** `StructuredRaceJob` creates `innerCancellationTokenSource = CancellationTokenSource.CreateLinkedTokenSource(...)` and never disposes it. The theory was that the parent `Context.CancellationTokenSource` keeps the linked source's callback registered, so over many evaluations the registrations accumulate unboundedly, causing a memory-exhaustion Denial of Service.
**Why this is not a vulnerability:** The parent is NOT long-lived. `Context.CancellationTokenSource` is refcounted by evaluation depth and set to `null` the moment the root evaluation tree unwinds (`Context.EnterEvaluationScope`/`ExitEvaluationScope`; the field is nulled at depth 0). Each `ConcurrentRace` evaluation creates a fresh parent source and a fresh linked source; when `Get()` returns, the parent field is nulled and the only references between the parent and the undisposed linked source are to each other — the owning `StructuredRaceJob` and its tasks are already garbage. That is an unreachable object island, and the .NET tracing GC reclaims islands regardless of internal cross-references, so both are collected together. There is no unbounded accumulation. The classic linked-CTS registration leak requires a parent token that lives for the process/request lifetime, which never happens here.
**Guidance for future scans:** Do not flag `CreateLinkedTokenSource` usages (or "undisposed `CancellationTokenSource`") whose parent is the per-evaluation, refcounted `Context.CancellationTokenSource`. Their lifetime is bounded by a single evaluation tree and they form a GC-collectable island. Passing such a source to `IStructuredResourceGroup.AddResource(...)` for deterministic disposal is acceptable as minor hygiene, but it is NOT a HIGH-severity issue and must not be reported as a memory leak or DoS.
